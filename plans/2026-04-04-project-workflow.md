# 建立项目进度与协作规则持久化机制

## Status
done

## Summary
在仓库中增加一套固定的项目管理文件，解决“新会话不记得当前进度”和“协作约定只存在于对话里”的问题。目标是让后续每次开始新任务时，都能通过读取仓库文件恢复上下文，而不依赖会话记忆。

## Key Changes
- 新增 `PROJECT_RULES.md`
  - 记录长期协作规则和固定工作方式
  - 明确规定：每次新增功能前，必须先在 `plans/` 下创建计划文件
  - 明确规定：每次开始新会话时，优先读取规则文件、当前状态文件和最近计划文件
- 新增 `CURRENT_STATUS.md`
  - 记录项目当前状态，而不是历史计划
  - 至少包含：当前已完成内容、当前阶段、下一步目标、最近完成的计划、已知待办
- 规范 `plans/` 的职责
  - `plans/` 只存每次小功能或阶段功能的计划说明
  - 每个计划文件聚焦一个任务，不承担“全局当前状态”的职责
  - 后续计划文件继续使用日期加主题的命名方式
- 建立固定流程
  - 新会话开始时：先读 `PROJECT_RULES.md`、`CURRENT_STATUS.md`、`plans/` 最近相关计划
  - 规划新功能时：先在 `plans/` 写计划
  - 功能完成后：更新 `CURRENT_STATUS.md`，必要时给计划文件补充完成状态

## Test Plan
- 新会话开始时，仅通过读取这几个文件，就能回答：
  - 当前项目做到哪一步了
  - 最近完成了哪些功能
  - 下一步准备做什么
  - 我们有哪些固定协作规则
- 新增一个新功能时，能够先创建计划文件，再实现，再回写状态，不出现流程遗漏
- 检查规则与状态分工是否清晰：
  - `PROJECT_RULES.md` 不写临时任务细节
  - `CURRENT_STATUS.md` 不堆积历史计划全文
  - `plans/` 不承担全局状态主页作用

## Assumptions
- 这套机制优先服务于当前仓库内的持续协作，不追求通用项目管理模板
- `CURRENT_STATUS.md` 作为单一最新真相来源，在每次阶段性工作完成后及时更新
- 如果后续任务变多，再考虑给 `plans/` 增加索引文件；当前阶段先不引入额外复杂度
